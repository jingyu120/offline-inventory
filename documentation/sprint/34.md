### Sprint 34: Inventory Quarantine & Sales Accountability Architecture

Execute the following database and logic overhauls to support a strict Accountability and Approvals model. Adhere to Drizzle ORM standards and the offline-first paradigms in `ai-instructions.md`.

**1. Frictionless Quarantined Intake (Backend & Mobile):**

- **Backend:** Update the `InventoryLedger` / `Products` schema. Introduce an `inventory_status` enum with states: `PENDING_APPROVAL`, `AVAILABLE`, `COMMITTED`, `SOLD`.
- **Mobile:** Update `IntakeScreen.tsx`. Allow warehouse staff to log incoming stock instantly without blockers. Hardcode the outgoing sync payload to set the newly registered inventory strictly to `PENDING_APPROVAL`. Update `AvailableItemsSelector.tsx` to explicitly filter out any items that are not `AVAILABLE` so reps cannot sell unapproved stock.

**2. Transaction Accountability Matrix (Backend & Mobile):**

- **Backend:** Update the `Orders` and `AuditEvents` Drizzle schemas. Expand the relational tracking fields to strictly enforce: `shop_id` (Client), `executed_by_id` (User who tapped submit), `salesperson_id` (User owning the sale), and `approved_by_id` (Nullable until manager review).
- **Mobile:** Update the checkout cart execution flow. Inject the mocked `actor_id` generated in Sprint 33 into both the `executed_by_id` and `salesperson_id` fields for local drafts, anticipating the future authentication gateway.

**3. Dual Approval Queues (Admin Dashboard):**

- Target the web/admin views (e.g., `features/admin/`).
- Build a `PendingIntakeApproval.tsx` table component. It must query all inventory with `inventory_status = 'PENDING_APPROVAL'`, allowing a manager to click "Authorize" to flip the stock to `AVAILABLE`.
- Build a `PendingSalesApproval.tsx` table component. When an order requires a margin override or hits a credit limit, route it here so a manager can populate the `approved_by_id` field and release the order.

**4. Lightweight Inbound Manifest (Optional Transit Feature):**

- **Backend:** Create a simple `ExpectedInbound` schema tracking `sku`, `expected_quantity`, `origin` ('Thailand'), and `estimated_arrival_date`.
- **Mobile:** Create an `InboundForecastList.tsx` component inside the inventory UI. This provides a read-only list for the intake team so they know what trucks to expect today, without locking them into rigid transit tracking rules.
